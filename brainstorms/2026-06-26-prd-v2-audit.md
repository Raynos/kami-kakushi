### Blocking defects (confirmed; each = a human design pick → see questions)
- **B1 — Name pillar T2 tie-out is broken; the pillar auto-clears the v1 end-gate with ZERO deeds.** `JUDGE_K[Name][T2]=42,000` is computed off a stale `TIER_REF_NAME=140K`, not Name's own 28K good band — so the seasonal stream *alone* (42K) is 150% of the good band (= its GREAT band), while §4.2.1/§4.8.1b itemize **zero** Name deeds. The basis clamps `f_Name→1.0` once the other three pillars merely reach good, so "good in Name" is **vacuous** at the climactic gate. Falsifies three locked statements (seasonal=30%-of-good; deeds-70/seasonal-30; breadth-floor/deeds-dominate) and the §6.6.1 verifier can't pass for Name without an unstated exemption. Also §4.2.2 "Name gates no v1 tier" flatly contradicts §4.1. → **Q6**.
- **B2 — Estate builds E1/E2/E3 are gated on pillar-Influence floors that are structurally unreachable where the builds sit.** E1@R4 needs Estate≥0.3K and E2@R6 needs Estate≥0.6K+Arms≥0.3K — but those are T0 Phase-1 rungs where pillar Influence is pinned at 0 (Phase-2-only accrual, FU7). E3@G4 needs Estate≥40K+Arms≥15K but at G4 pillars are frozen at end-T1 (≤15K/≤10K) — impossible by ~2.7×. The §4.7.5 affordability tie-out checks only the koku *cost* and skips the Influence floors. → **Q7**.
- **B3 — Save import-over can be silently discarded by a higher-counter autosave.** An imported file (old self-export / QA hand-off / friend's save) carries its own typically-*lower* `saveCounter`; the spec never says import rebases the counter above local-max nor rewrites all backends, so on next load the still-present autosave (higher counter) is elected and the player-confirmed import is **silently lost** — and the multi-backend redundancy *propagates* the loss. Related: additive optional-fields *without* a `schemaVersion` bump (§6.8.2) lets a stale-CDN build silently truncate a newer same-version save. Violates the save layer's "no silent loss" headline. → see Q (save) in master list.

**Critical**

**High**

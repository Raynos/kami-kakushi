### F1 · DEV panel renders a big white block — ✅

**Verbatim:** _"The main page has a big white block because of the dev UI. white
looks ugly, make it the same background everywhere."_
**Reading:** The DEV panel's container background is white/unset instead of
inheriting the game's dark background, so a big white block runs down the whole
right strip and clashes with the woodblock/ink look. Make it match the game
background so there's no white seam. This is a DEV-only surface, but the human
still wants it visually consistent with the game.
**Fix plan (code):** Find the DEV panel component/CSS and set its container/body
background to the game's canonical dark background variable — reuse the existing
`--bg` / ink variable, don't hardcode a colour. (A parallel subagent is applying
this fix now.)
**Doc-update plan:** Add a rule to
[`ui-design.md`](../../docs/living/ui-design.md) (the woodblock/ink UI bible)
that DEV/tooling surfaces must inherit the game's dark background — no
white/browser-default backgrounds anywhere, ever, even on dev-only chrome.
**Distilled rule:** No raw white / browser-default backgrounds anywhere in the
app, including DEV-only tooling and chrome. Every surface inherits the game's
dark ink background by default. White reads as unfinished/ugly and breaks the
woodblock immersion. → § Taste distillation (home: `ui-design.md`).
**Fixed in:** `7642fd7`

### F2 · ?dev=no opt-out + dev panel is shoving the UI off-center — ✅

**Verbatim:** _"I want to be able to open http://localhost:5173/?dev=no and just
not see the dev panel at all. I have a suspicion that the UI is off center
because of the position of the dev panel and that if the dev panel is fully
removed / never rendered / fully hidden that the UI would be different."_
**Reading:** Two linked asks. (1) A `?dev=no` URL query param that FULLY
suppresses the DEV panel — not rendered at all / never mounted in the DOM, not
merely visually hidden. (2) The human suspects the main game UI is off-center
because the DEV panel occupies real layout space (it's a right-hand column in
the layout flow, so the game content is centered in the leftover space, not the
full viewport). Verify that hypothesis: if true, the DEV panel should be a
fixed/overlay element outside the centering layout so the game UI is always
centered on the FULL viewport whether or not the panel is present. The `?dev=no`
case and the off-center fix are related — with the panel gone the layout should
be identical to the panel being an overlay.
**Fix plan (code):** (a) Add a `?dev=no` (or `dev=0`/`dev=false`) query-param
check that gates DEV-panel mounting — when set, the panel is never rendered.
Keep the existing default (panel shows in DEV builds). (b) Investigate whether
the DEV panel is part of the layout flow (e.g. a flex/grid column) that
de-centers the game; if so, make the panel `position: fixed`/overlay so the game
UI centers on the full viewport regardless. Confirm by comparing the layout with
panel present vs `?dev=no`. (Fix agent will be dispatched once the F1 CSS fix
lands, to avoid a same-file race.)
**Doc-update plan:** Add to
[`ui-design.md`](../../docs/living/ui-design.md): the game UI must center on the
FULL viewport independent of any DEV/tooling chrome (dev tooling is an overlay,
never in the centering layout); document the `?dev=no` opt-out as the way to
preview the true player-facing layout. If `?dev=no` becomes a supported dev
affordance worth locking, consider a short note in
[`qa-playtesting.md`](../../docs/living/qa-playtesting.md) too.
**Distilled rule:** DEV/tooling chrome must never affect the player-facing
layout. The game UI centers on the full viewport as if the dev panel doesn't
exist; dev tooling is a fixed overlay, not a layout-flow element. Always provide
a way to preview the true player layout (e.g. `?dev=no`). → § Taste distillation
(home: `ui-design.md`, with a QA note in `qa-playtesting.md`).
**Fixed in:** `7642fd7`

### F3 · "Open your eyes" button not centered in the intro card — ✅

**Verbatim:** _"Open your eyes button is not centered."_
**Reading:** In the cold-open intro card, the title/subtitle/prose all center on
the card's axis but the "Open your eyes" button is left-aligned, breaking the
centered composition. The button should be centered on the same axis as the rest
of the card's content.
**Fix plan (code):** In [`src/ui/styles.css`](../../src/ui/styles.css) (single
stylesheet) find the intro/cold-open card's button (and/or its container) and
center it horizontally — likely the button's wrapper needs `text-align: center`
or the flex container needs `align-items: center` / `justify-content: center` /
`margin: 0 auto`. Match how the title/prose are centered. This is being batched
with the F2 layout fix (same file, same layout concern) into one fix agent to
avoid a same-file race.
**Doc-update plan:** Add to
[`ui-design.md`](../../docs/living/ui-design.md): in a centered card/panel, a
single primary action (button) inherits the card's center axis — don't let it
default to left-aligned. (This can share the F2 ui-design.md layout section.)
**Distilled rule:** In a centered card/panel composition, a single primary
action inherits the card's center axis — a lone button should be centered with
the rest of the content, not left-aligned by default. → § Taste distillation
(home: `ui-design.md`).
**Fixed in:** `7642fd7`

### F4 · Main UI too narrow + DEV panel de-centers it — ✅

**Verbatim:** _"The main UI component is too narrow, too much whitespace left &
right ( could be caused by dev panel ). The main UI component should take up more
of the screen. the dev panel should be a floating panel that's just abstractly on
top somewhere, and when collapsed should be as small as possible, maybe anchored
to the bottom somewhere."_
**Reading:** The DEV panel mount reserved a 16rem right gutter on `#app`
(`app.style.paddingRight`), which shoved the game UI left, created the L/R
whitespace, and (with `body` having no background) exposed a white strip — the
same root cause as F1/F2. The main UI (`.shell`, capped at max-width 980px)
should be wider and take more of the screen. The DEV panel should be a true
floating overlay (position:fixed, reserving NO layout space), bottom-anchored,
and collapse to as small as possible.
**Fix plan (code):** Delete the `#app paddingRight:16rem` gutter in
[`src/ui/dev.ts`](../../src/ui/dev.ts); make the panel bottom-anchored +
`width:fit-content` when collapsed (grow to 15rem expanded); widen `.shell`
max-width 980→1200 and give `body` a dark `--ink` background so margins read as
an intentional woodblock border, not white. (Being implemented now in the main
session as part of the full-viewport app-shell restyle.)
**Doc-update plan:** [`ui-design.md`](../../docs/living/ui-design.md) — the main
UI is a centered paper column (~1200px) on an ink ground that fills the viewport;
DEV/tooling chrome is a fixed overlay reserving no layout space (ties to F2's
rule).
**Distilled rule:** The player UI fills the viewport as a centered paper column
on the ink ground — no cramped narrow column, no raw-white margins. DEV/tooling
chrome is a fixed floating overlay that reserves ZERO layout space and collapses
to minimal. → § Taste distillation (home: `ui-design.md`).
**Fixed in:** `7642fd7`

### F5 · Adopt a fixed header + fixed footer app-shell — ✅

**Verbatim:** _"Other games have a fixed footer on the bottom for UI elements, I
think we would benefit from the fixed header and fixed footer."_
**Reading:** Model the layout like other idle/RPG games: a fixed header pinned to
the top and a fixed footer pinned to the bottom, with the game content filling
the space between. The footer hosts persistent UI controls. (Reference
screenshots the human shared show idle-RPG shells with a header status row and a
footer holding save/settings/version controls.)
**Fix plan (code):** Make `.shell` a full-height flex column (`height:100dvh;
overflow:hidden`): header rows (titlebar/vitals/nav) `flex:0 0 auto` pinned top;
`.workspace` `flex:1; min-height:0` fills the middle; add a new
`<footer class="appbar-footer">` pinned bottom holding the version stamp + the
Settings button (moved out of the titlebar). (Being implemented now in the main
session as part of the full-viewport app-shell restyle.)
**Doc-update plan:** [`ui-design.md`](../../docs/living/ui-design.md) — document
the app-shell contract: fixed header (identity + vitals + nav) / flexible content
/ fixed footer (persistent controls: version, Settings, …).
**Distilled rule:** The game uses a fixed-header / flexible-content /
fixed-footer app shell (like established idle-RPGs). Persistent controls live in
the footer; identity + vitals live in the header; the content region fills the
space between and scrolls internally. → § Taste distillation (home:
`ui-design.md`).
**Fixed in:** `7642fd7`

### F6 · No page scrollbar — size the UI to the viewport — ✅

**Verbatim:** _"There's a damn scroll bar in the main UI, like the UI is not sized
vertically based on the available space. Of course it needs a minimum size, but it
should not have any scroll bars"_
**Reading:** The whole page scrolls because the content (a full-height header plus
a `.log` capped at `max-height:78vh`/`min-height:60vh`) exceeds the viewport.
Instead the app shell should be exactly viewport-height with NO page scrollbar;
the individual content panes (story log, work column) scroll INTERNALLY within the
available space. A minimum size is fine, but no page-level scrollbars.
**Fix plan (code):** `body`/`#app` `overflow:hidden` + `height:100dvh`;
`.workspace` a full-height grid row (`grid-template-rows:minmax(0,1fr);
min-height:0; overflow:hidden`); the `.log` and `.work` panes get `min-height:0;
overflow-y:auto` (drop the `60vh/78vh` caps) so they scroll internally. A
`@media (max-width:720px)` fallback relaxes to natural page flow so mobile still
works. (Being implemented now in the main session as part of the full-viewport
app-shell restyle.)
**Doc-update plan:** [`ui-design.md`](../../docs/living/ui-design.md) — the app
shell is exactly viewport height with no page scroll; content panes scroll
internally; a narrow-screen fallback allows page flow.
**Distilled rule:** The app shell is sized to the viewport (100dvh) with NO page
scrollbar; content panes scroll internally within the available height. Respect a
minimum size and fall back to natural page flow on narrow screens. → § Taste
distillation (home: `ui-design.md`).
**Fixed in:** `7642fd7`

## Taste distillation — the general rules (graduate to living docs)

> The point of this section: turn one-off feedback into a **standing rule** a
> future implementer will read _before_ building, so the taste is applied by

  surface in the app inherits the game's dark ink background by default,
  including DEV-only tooling and chrome. White reads as unfinished/ugly and
  breaks the woodblock immersion. **Home doc:**

  centered card/panel composition, a single primary action (button) should be

## Taste distillation — the general rules (graduate to living docs)

> The point of this section: turn one-off feedback into a **standing rule** a
> future implementer will read _before_ building, so the taste is applied by
> default. Each entry names its **home doc** and moves there once settled.

- **No raw white / browser-default backgrounds anywhere** (from F1) — every
  surface in the app inherits the game's dark ink background by default,
  including DEV-only tooling and chrome. White reads as unfinished/ugly and
  breaks the woodblock immersion. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **DEV/tooling chrome must never affect the player-facing layout** (from F2) —
  the game UI centers on the full viewport as if the dev panel doesn't exist;
  dev tooling is a fixed overlay, not a layout-flow element. Always provide a way
  to preview the true player layout (e.g. `?dev=no`). **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) (with a QA note in
  [`qa-playtesting.md`](../../docs/living/qa-playtesting.md)).
- **A lone primary action inherits its card's center axis** (from F3) — in a
  centered card/panel composition, a single primary action (button) should be
  centered with the rest of the content, not left-aligned by default. **Home
  doc:** [`ui-design.md`](../../docs/living/ui-design.md).
- **The player UI fills the viewport as a centered paper column** (from F4) — no
  cramped narrow column, no raw-white margins; the game UI is a centered paper
  column on the ink ground. DEV/tooling chrome is a fixed floating overlay that
  reserves ZERO layout space and collapses to minimal. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **Fixed-header / flexible-content / fixed-footer app shell** (from F5) — the
  game uses an established idle-RPG shell: persistent controls live in the footer,
  identity + vitals live in the header, and the content region fills the space
  between and scrolls internally. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **The app shell is sized to the viewport — no page scrollbar** (from F6) — the
  shell is exactly viewport height (100dvh) with NO page scrollbar; content panes
  scroll internally within the available height. Respect a minimum size and fall
  back to natural page flow on narrow screens. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **Narrative/log content scrolls smoothly to follow the newest entry** (from
  F7) — log/narrative updates scroll SMOOTHLY to the newest line (honoring
  reduced-motion) rather than an instant jump; the motion of the story arriving
  is part of the feel. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) (feel nod in
  [`fun-factor.md`](../../docs/living/fun-factor.md)).
- **The message/story log lives in the RIGHT-hand column** (from F8) — the
  established idle-RPG convention: the message/story log occupies the right
  column while the interactive work/actions column sits on the left (supersedes
  the earlier v0.3.1 story-log-as-left-hero placement; live human taste is canon,
  D-022). **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **The event log is filterable by channel via a bottom tab/toggle bar** (from
  F9) — Story / Work / Combat / Progression / … channels toggle what the log
  shows. The STORY is a first-class, returnable tab: the narrative is content
  worth revisiting, not ephemeral scrollback; work/actions noise is the least
  important channel. New UI surface → D-075 diverge applies. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) (story-as-keepable-read nod in
  [`fun-factor.md`](../../docs/living/fun-factor.md)).
- **The UI is COMPACT and dense like an established idle-RPG** (from F10) — a
  tight type scale, small control heights, minimal chrome padding. Chrome
  (header/footer/tabs/resource row) must stay minimal; the content (log +
  actions) gets the space. Bigger-is-not-better. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **Lay information across multiple distinct panels (idle-RPG dashboard
  density)** (from F11) — not a couple of crammed columns; reconciled with
  reveal-based progression so panels appear as their surface unlocks. **Home
  doc:** [`ui-design.md`](../../docs/living/ui-design.md) (pending a plan/diverge
  to scope it).
- **Story/narration text reveals slowly, GBA typewriter-style** (from F12) — not
  instant, not a fast cascade; scoped to the story channel, never combat spam;
  reduced-motion → instant. Slow text is part of the atmosphere. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) (feel nod in
  [`fun-factor.md`](../../docs/living/fun-factor.md)).
- **Character dialogue is incremental and interactive** (from F13) — the intro
  (and NPC dialogue generally) advances beat by beat and the player can respond /
  talk back, not read a wall of text. NPCs the player meets are people to answer,
  not narration. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) (+ an ADR once the direction
  locks).
- **Atmospheric/ceremonial surfaces keep generous breathing room** (from F14) —
  the cold-open, seals, and similar ceremonial cards stay roomy; the
  compact-density rule is for the game chrome, not these. The opening reveals
  slowly with a delayed CTA (the slow-waking beat), not all-at-once. **Home
  doc:** [`ui-design.md`](../../docs/living/ui-design.md).
- **Reveal/animation beats must fire when the player is watching** (from F15) —
  after the gating interaction, never pre-run behind a curtain; an unseen
  animation is a wasted beat. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **(dev-ergonomics, not player taste) The DEV panel splits Settings vs
  Variants** (from F16) — variants are recency-ordered (newest on top)
  collapsible summaries that expand to their options, so a long variant list
  stays reviewable. **Home:** the `diverge` skill notes / DEV tooling (not
  `ui-design.md`).
- **(dev-ergonomics, not player taste) Approved variant wins → strip the rest;
  variants carry stable V-numbers** (from F17) — when the human approves a
  variant it becomes the prod default and the unpicked alternates are removed
  immediately (D-075 zero flag-debt); don't let dead variants linger. Every
  variant carries a stable, unique V-number (V0…VN) for quick human reference.
  **Home:** the `diverge` skill notes / dev-ergonomics (not `ui-design.md`).
- **(dev-ergonomics, not player taste) DEV variant selections round-trip through
  the URL query params** (from F18) — picking a variant writes
  `?<surface>=<variant>` to the URL; load reads the params and applies them, so a
  refresh preserves the review state (and the URL is shareable). **Home:** the
  `diverge` skill notes / dev tooling (not `ui-design.md`).
- **Story/narration lines reveal with a real character-by-character typewriter**
  (from F19) — a genuine GBA-feel typewriter, not a fast line-flash; scoped to the
  story channel (never combat spam); reduced-motion → instant. (Sharpens F12.)
  **Home doc:** [`ui-design.md`](../../docs/living/ui-design.md) (feel nod in
  [`fun-factor.md`](../../docs/living/fun-factor.md)).
- **Filtered channels surface an unread indicator** (from F20) — a filter tab
  shows a badge/dot when new content lands in a channel you're not viewing;
  switching to it clears the mark. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **Work-stamina and health are distinct resources with distinct recovery
  actions** (from F22) — labour drains/refills work-stamina; combat drains/heals
  health; each has its own meter and its own recovery action, and the two rests
  are never conflated. A CORE game-mechanics change, not UI polish. **Home doc:**
  [`prd.md`](../../docs/living/prd.md) + an ADR in
  [`decisions.md`](../../docs/living/decisions.md) once locked.
- **The story log is legible by VOICE** (from F23) — narrator prose, NPC speech,
  and player speech each read distinctly (colour + emphasis), never a flat
  undifferentiated wall. Full three-way distinction wants speaker-tagged log
  entries (a small core/content model); player speech arrives with the interactive
  intro (F13). Part of the story-presentation cluster (F12, F13, F15, F19, F23).
  **Home doc:** [`ui-design.md`](../../docs/living/ui-design.md) (§5.1 event-log).
- **New Game / state reset returns the UI to its zero state too** (from F25) — a
  reset restores the default tab (Work) and default log filter (Story), not just
  the game state; the renderer's UI closure vars are reset alongside the game
  state so the player lands on the zero state. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **Speech is coloured by the speaker's character CATEGORY** (from F26) — a small
  palette of voice-colours (household/steward, physician, official, villager,
  player, narrator, …) so WHO is talking reads at a glance, not just that someone
  is; narration and each character-category read distinctly. Needs speaker-tagged
  dialogue (sharpens F23's single speech colour, which is a stopgap). Part of the
  intro/dialogue cluster. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) (§5.1) + the dialogue/intro
  design.
- **A scrolling region under a fixed header separates cleanly** (from F28) — a
  header border + a soft top fade so content never bleeds half-readable under the
  header. **Home doc:** [`ui-design.md`](../../docs/living/ui-design.md).
- **Modal/dialog surfaces keep generous internal padding** (from F29) — padding +
  section spacing on modals; the compact chrome-density rule is for the game HUD,
  not modals. **Home doc:** [`ui-design.md`](../../docs/living/ui-design.md).
- **The a11y text-scale changes TEXT size only** (from F30) — multiply the type
  tokens (`--fs-*`), not the root rem unit, so layout/spacing stays put; scaling
  text must not reflow the whole UI. **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md) §8 (responsive/a11y).
- **The settings modal is organised into sub-tabs (Settings / Saves / About)**
  (from F31) — not one long column; the save UI is a streamlined manage-saves
  surface (absorbs F29's padding). **Home doc:**
  [`ui-design.md`](../../docs/living/ui-design.md).
- **A loaded save starts IDLE** (from F32) — transient "currently auto-doing X"
  state (auto-labour, auto-rake, auto-fight targets) is reset on load so a
  refresh never resumes auto-ing; the player opts back in. Persist progress, not
  in-flight automation. **Home doc:**
  [`prd.md`](../../docs/living/prd.md) (persistence) /
  [`ui-design.md`](../../docs/living/ui-design.md).

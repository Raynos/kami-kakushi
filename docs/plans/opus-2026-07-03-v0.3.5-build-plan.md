## Who builds this — Fable or Opus?

**Opus** (per D-124: subagents inherit the parent's model). These are pure-core +
reducer + render changes with real mechanical judgment (storage-cost tuning, verb
re-homing) and must keep the append-only render contract + the invariant tests
green. No new UI *surface* is designed from scratch (so no `diverge` mandate) —
the Quests tab and the wall-weapon token are re-homings of existing content, not
net-new visual language. All seven have **locked** specifics, so this is safe to
hand to a build subagent.

## The seven tasks (each a signed decision)

Ordered roughly by isolation (cheap/independent first). Each names its **source
decision**, the **files** it most likely touches, and its **done-when**.

### 1 · Rice storage costs something (store-vs-sell fix)

- **Decision:** the seasonal store-vs-sell choice is dominated — kura storage is
  free, lossless, loss-safe → "always hold until spring" wins. Make holding cost
  something. (`Store-vs-sell → Make holding cost something`.)
- **Approach (LOCKED — human, 2026-07-03):** **both levers.** **(a) Spoilage** — a
  per-season decay applied to **all rice, carried AND stored** (pure hoarding
  always bleeds; you can't out-store the loss). **(b) An upgradeable kura capacity
  cap** — the kura holds only N rice, and **raising N is an estate/kura upgrade**
  (a coin sink + a reason to invest). Magnitudes liquid, tuned by feel (D-059).
- **Files:** `src/core/content/balance.ts` (spoilage rate + base/upgraded cap
  constants), the season-tick reducer in `src/core/intents.ts` (apply decay on
  season change to carried + banked rice), the kura-upgrade path (cap raise),
  `src/ui/render.ts` (show the cap `stored/N` + spoilage).
- **Done-when:** a balance-sim / manual play shows sell-now and hold-for-spring
  are both viable depending on need — not one dominant line; rice visibly spoils
  in hand and store; the cap is a real wall you can upgrade past. The
  `economy-balance-watch` item closes.

### 2 · Now-view migrates to append-only

- **Decision:** `renderNowView` still does a full `textContent=''` rebuild
  (`render.ts:3674`) — the one surface the append-only engine didn't reach. Fix it
  too. (`Now-view residual → Fix it too`.)
- **Files:** `src/ui/render.ts` (renderNowView), reuse `src/ui/reconcile.ts`.
- **Done-when:** a zero-churn/node-identity test covers the Now-view like every
  other migrated surface; idle re-render produces no DOM churn there.

### 3 · Stagger the Inventory tab reveal to R3

- **Decision:** Map + Estate + Inventory all light at R1 (a triple-reveal). Hold
  Inventory to R3 so R1 isn't a slam of three new tabs.
  (`R1 reveal → Stagger Inventory to R3`.)
- **Files:** the tab-gate predicate in `src/ui/render.ts` (`tabHasContent` /
  `visibleTabs`, ~`:1121-1148`) — gate Inventory on `tab-combat`/R3 instead of
  `panel-estate`/R1.
- **Done-when:** at R1 only Map + Estate reveal; Inventory appears at R3. The
  reveal-order render test is updated to assert the new sequence (and could have
  gone RED on the old gate).

### 4 · Quests get their own dedicated tab

- **Decision:** override — quests were folded into Character as "Undertakings 用";
  you want them back as a **dedicated tab** (the set becomes **7**).
  (`Quests home → Own Quests tab`.)
- **Reveal cadence (human, 2026-07-03 — D-119):** the Quests tab reveals at
  **R5**, its **own** quest-log beat — **not** grouped into the R3 combat wave.
  So the full cadence is Work R0 · Map + Estate R1 · Character R2 · Combat +
  Inventory R3 · **Quests R5**. Gate the Quests tab on its R5 content-unlock, not
  on `tab-combat`/R3.
- **Files:** the `Tab` type + `TAB_LABEL`s (`render.ts:247`, `:1088`), the
  reveal-gate list (Quests gated at R5), `renderQuests` re-homed to its own tab
  body, and the Character tab loses the Undertakings section. Glyph 用 (provisional
  — a taste call).
- **Done-when:** a Quests tab is navigable, **revealed at R5** (not the R3 wave);
  Character no longer shows Undertakings; the reveal-order render test asserts
  Quests appears at R5 (and could have gone RED on an R3 gate); render tests
  updated for 7 tabs.

### 5 · Hearth homes the cook verb

- **Decision:** override — the hearth shipped as a raw `body`/satiety +12 bonus;
  it should **home the `cook_meal` verb** (diegetic, not a stat).
  (`Hearth → Home the cook verb`.)
- **Files:** `src/core/content/home.ts` (drop the hearth's `body` comfort; make it
  the cook locus), `src/core/intents.ts` (`cook_meal` gated on / surfaced at the
  hearth), `src/ui/render.ts` (cook affordance shown in the home/hearth context).
- **Done-when:** cooking happens *at* the hearth in the running game; the hearth
  is no longer a flat stat bump. Update `home.test.ts` accordingly.

### 6 · Chest becomes real storage

- **Decision (LOCKED — human, 2026-07-03):** override — the chest (nagamochi)
  shipped as a +5 satiety stat; it becomes a **belongings-only, modest buffer**
  (keepsakes/belongings — **not** rice/goods; keep it prestige-flavoured and out
  of the rice economy so it doesn't tangle with the kura cap/spoilage in #1).
  Pairs with **no** morale system (housing stays rest / satiety / **storage**).
- **Files:** `src/core/content/home.ts` (chest defines a small belongings
  capacity, not a `body` comfort), the belongings model in `src/core/state.ts` if
  a buffer needs backing state, `src/ui/render.ts` (show what's kept).
- **Done-when:** the chest holds **belongings** with a real, modest capacity a
  player can see and use; it does NOT store rice/goods; the prestige-not-power
  invariant still holds (storage isn't a combat stat).

### 7 · koku Phase 5 (T0 slice) — one home status token + status-mirror

- **Decision (LOCKED — human, 2026-07-03):** koku Phase 5 (the full surname → two
  swords → gōshi ladder) is split out to T1–T5 planning. **T0's single token is a
  WEAPON MOUNTED ON THE WALL** — specifically **the weapon you wield when you reach
  R5** (carrying-pole, or whatever you've crafted/found), granted at **R5** and
  shown by the housing **status-mirror**. **Explicitly NOT a surname** — the
  surname is the **T3 origin-story** beat (you already have one); and it must be
  *your* weapon, never a random sword you have no skill with.
- **Files:** `src/core/ranks.ts` (`rewardOnReach` at R5 grants the token, reading
  the wielded weapon), `src/core/content/home.ts` (the status-mirror hook —
  deferred seam at `~:177` — renders the mounted weapon), `src/ui/render.ts` (show
  it on the home wall).
- **Done-when:** reaching **R5** mounts your currently-wielded weapon on the home
  wall (updates to reflect the actual weapon, never a generic sword); no attempt
  at the full surname→swords→gōshi ladder (that's T1–T5).

## Sequencing note

Seven tasks, all with **locked** specifics — safe to hand to a single build
subagent. 1–3 are isolated and cheap (independent gates/surfaces). 4 (Quests tab,
now revealing at **R5**) is the biggest IA change (the PRD/ui-design ripple for
the 7-tab set already landed). 5–6 (housing: hearth→cook, chest→belongings-store)
pair naturally. 7 (the R5 wall-weapon token) touches the rung/reward seam. Ship
per-task with the full `verify` gate green each time; each override updates the
test that was asserting the old default.

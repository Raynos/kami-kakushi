# v0.3.5 build plan — the eight code deltas from the agent-default audit

**Status:** 📋 PROPOSED — awaiting human read; **no code changed yet**. Authored
2026-07-03 from the interactive agent-default audit
([`project/audit/reports/2026-07-03-agent-default-decision-audit.md`](../../project/audit/reports/2026-07-03-agent-default-decision-audit.md)).
Every task below is a human-signed decision from that walk-through — either an
**override** of a shipped agent default, or newly-scoped build work. The
doc-side ripples (PRD, ADRs, ui-design) live in the companion
[`opus-2026-07-03-v0.3.5-cleanup-docs.md`](opus-2026-07-03-v0.3.5-cleanup-docs.md).

## Who builds this — Fable or Opus?

**Opus, all eight.** These are pure-core + reducer + render changes with real
mechanical judgment (storage-cost tuning, a capstone branch, verb re-homing) and
must keep the append-only render contract + the invariant tests green. No new UI
*surface* is being designed from scratch (so no `diverge` mandate) — the Quests
tab and status-token are re-homings of existing content, not net-new visual
language. Fable's lane is the companion cleanup-docs plan (PRD/taste), not this.

## The eight tasks (each a signed decision)

Ordered roughly by isolation (cheap/independent first). Each names its **source
decision**, the **files** it most likely touches, and its **done-when**.

### 1 · Rice storage costs something (store-vs-sell fix)

- **Decision:** the seasonal store-vs-sell choice is dominated — kura storage is
  free, lossless, loss-safe → "always hold until spring" wins. Make holding cost
  something. (`Store-vs-sell → Make holding cost something`.)
- **Approach (pick during build, tune by feel — D-059):** the lightest lever that
  makes the decision live — a per-season **spoilage** %, a **capacity cap** on the
  kura, or a small **coin upkeep**. Spoilage is the most legible (rice visibly
  rots if hoarded) and needs no new UI.
- **Files:** `src/core/content/balance.ts` (a spoilage/cap constant), the
  season-tick reducer in `src/core/intents.ts` (apply decay on season change),
  `src/ui/render.ts` (show the kura's spoilage/cap state).
- **Done-when:** a balance-sim / manual play shows sell-now and hold-for-spring
  are both viable depending on need — not one dominant line. The
  `economy-balance-watch` item closes.

### 2 · Now-view migrates to append-only

- **Decision:** `renderNowView` still does a full `textContent=''` rebuild
  (`render.ts:3674`) — the one surface the append-only engine didn't reach. Fix it
  too. (`Now-view residual → Fix it too`.)
- **Files:** `src/ui/render.ts` (renderNowView), reuse `src/ui/reconcile.ts`.
- **Done-when:** a zero-churn/node-identity test covers the Now-view like every
  other migrated surface; idle re-render produces no DOM churn there.

### 3 · Stagger the Inventory tab reveal to R3

- **Decision:** Map + Estate + Inventory all light at R1 (a triple-reveal). Hold
  Inventory to R3 so R1 isn't a slam of three new tabs.
  (`R1 reveal → Stagger Inventory to R3`.)
- **Files:** the tab-gate predicate in `src/ui/render.ts` (`tabHasContent` /
  `visibleTabs`, ~`:1121-1148`) — gate Inventory on `tab-combat`/R3 instead of
  `panel-estate`/R1.
- **Done-when:** at R1 only Map + Estate reveal; Inventory appears at R3. The
  reveal-order render test is updated to assert the new sequence (and could have
  gone RED on the old gate).

### 4 · Quests get their own dedicated tab

- **Decision:** override — quests were folded into Character as "Undertakings 用";
  you want them back as a **dedicated tab** (the set becomes **7**).
  (`Quests home → Own Quests tab`.)
- **Files:** the `Tab` type + `TAB_LABEL`s (`render.ts:247`, `:1088`), the
  reveal-gate list, `renderQuests` re-homed to its own tab body, and the
  Character tab loses the Undertakings section. Pick a glyph for the Quests tab
  (a taste call — carry to cleanup-docs if unsure).
- **Done-when:** a Quests tab is navigable, gated on quest content unlocking;
  Character no longer shows Undertakings; render tests updated for 7 tabs.

### 5 · Hearth homes the cook verb

- **Decision:** override — the hearth shipped as a raw `body`/satiety +12 bonus;
  it should **home the `cook_meal` verb** (diegetic, not a stat).
  (`Hearth → Home the cook verb`.)
- **Files:** `src/core/content/home.ts` (drop the hearth's `body` comfort; make it
  the cook locus), `src/core/intents.ts` (`cook_meal` gated on / surfaced at the
  hearth), `src/ui/render.ts` (cook affordance shown in the home/hearth context).
- **Done-when:** cooking happens *at* the hearth in the running game; the hearth
  is no longer a flat stat bump. Update `home.test.ts` accordingly.

### 6 · Chest becomes real storage

- **Decision:** override — the chest (nagamochi) shipped as a +5 satiety stat; it
  should be a **storage buffer** where belongings sit. (`Chest → Make it real
  storage`.) Pairs with **no** morale system (housing stays rest / satiety /
  **storage**).
- **Files:** `src/core/content/home.ts` (chest defines storage capacity, not a
  `body` comfort), the belongings/resource model in `src/core/state.ts` if a
  buffer needs backing state, `src/ui/render.ts` (show what's stored).
- **Done-when:** the chest holds belongings/goods with a real capacity a player
  can see and use; the prestige-not-power invariant still holds (storage isn't a
  combat stat).

### 7 · R7 capstone becomes a mechanical branch

- **Decision:** override — the R7 capstone pick was a "light branch" (flag only);
  it should **matter mechanically**. (`Capstone weight → Make it matter`.)
- **Approach:** the R7 choice sets a genuine, legible mechanical fork carried into
  play (e.g. a lasting modifier or an unlocked path), not just a remembered flag.
  Scope it small but real — it lands at the T0→T1 seam.
- **Files:** `src/core/content/rungBeats.ts` (R7 option payloads), the state field
  the branch reads, wherever the consequence applies.
- **Done-when:** the two R7 picks produce a *different* mechanical outcome a
  player can feel; a test asserts the branch diverges (could have gone RED when it
  was flag-only).

### 8 · koku Phase 5 (T0 slice) — one home status token + status-mirror

- **Decision:** koku Phase 5 (status tokens: surname → two swords → gōshi) is
  split out of the economy plan. **T0 scope is deliberately tiny:** the player
  earns **one** hard-won home **status token** across R1→R7, and the housing
  **status-mirror** displays it. The **full status ladder is T1–T5 PRD planning**
  (see cleanup-docs), *not* built here.
- **Files:** `src/core/ranks.ts` (`rewardOnReach` grants the one token at its
  rung), `src/core/content/home.ts` (the status-mirror hook — deferred seam at
  `~:177` — renders the earned token), `src/ui/render.ts` (show it in the home).
- **Done-when:** reaching the chosen rung grants the single T0 status token and it
  visibly appears in the home; no attempt at the full surname→swords→gōshi ladder.

## Sequencing note

1–3 are isolated and cheap (independent gates/surfaces). 4 (Quests tab) is the
biggest IA change and should ride with the cleanup-docs PRD ripple for the 6→7
tab set. 5–6 (housing) pair naturally. 7–8 touch the rung/ceremony seam and can
share a pass. Ship per-task with the full `verify` gate green each time; each
override updates the test that was asserting the old default.

## 0 · Problem & shape

Every playtest, bug repro, QA screenshot run, and balance-tuning session
today starts from a cold boot plus a hand-driven climb (`__qa.toRung`,
`jumpToAscension`, manual fights). The interesting moments — the wolf
fight, the ascension ceremony, the post-loss slump, the broken-weapon
bind — each cost minutes of setup and are re-derived slightly
differently every time. This plan makes them one click: a small library
of NAMED fixture saves, committed to the repo, loadable from the DEV
panel and from `window.__qa.loadFixture(name)`, so "reproduce X" becomes
"load X, look".

The hard repo norm shapes everything: **fixtures are never
hand-maintained.** Each one is GENERATED by driving the REAL engine
through REAL intents from a fixed seed (the exact machinery
`t0-arc.test.ts` already uses), so `npm run fixtures:regen` rebuilds the
whole library from scratch at any time — after a balance retune, a
schema bump, or a content rename. The envelope carries the seeded RNG
cursors and the clock by construction, so determinism rides along free.

## 2 · Design

### 2.1 Where fixtures live + what format

**Home: `src/fixtures/`** —

```
src/fixtures/specs.ts        // hand-written CODE: the waypoint specs
src/fixtures/index.ts        // DEV registry: import.meta.glob the JSONs
src/fixtures/fixtures.test.ts// the round-trip + waypoint-assert test
src/fixtures/saves/<name>.json  // GENERATED envelopes (committed)
src/scripts/gen-fixtures.ts  // the generator (gen-docs idiom)
```

Inside `src/` because (a) vitest resolves there, (b) the DEV panel can
`import.meta.glob('./saves/*.json', { eager: true })` with zero
transport machinery, and (c) the pure-core boundary holds: `specs.ts`
imports only the core's public index (like `src/sim/` in the sim-gates
plan); nothing in `src/core` ever imports fixtures.

**Format: the raw envelope as pretty-printed JSON (2-space), one file
per fixture — not base64.** Rationale:

- **Reviewable regen diffs.** A regen after a balance tweak shows
  `"coin": 120` → `"coin": 96` as a line diff — exactly what the
  balance cockpit and a reviewing agent need. Base64 turns every regen
  into one opaque changed line; you can never see WHAT moved.
- **Re-encoding on load is one expression.** `__qa.loadFixture` does
  `importState(toBase64(JSON.stringify(env)))` — reusing the FULL
  import → migrate → validate path `__qa.load` already exercises. (Add
  `toBase64` to `src/persistence/index.ts` exports; it already exists
  in `codec.ts`.)
- **Diff noise is bounded and honest.** The noisy part is the log ring
  (≤ 300 entries, `LOG_RING_MAX`); it changes only when the driven arc
  changes — which is precisely when a human might want to skim what
  changed. Byte-stable regen (§2.2) means a no-op regen is a no-op diff.
- **Prettier:** add `src/fixtures/saves/` to `.prettierignore` — the
  generator is the SOLE formatter of these bytes (byte-stability
  requires exactly one writer; same contract as generated docs).

### 2.2 The generator — declarative specs, real intents, byte-stable

`src/fixtures/specs.ts` — a typed registry, the single source the
generator, the round-trip test, and the DEV panel all read:

```ts
export interface FixtureSpec {
  readonly name: string;   // 'fresh-R3-pre-wolf' — file + panel + __qa key
  readonly blurb: string;  // the one-line panel description
  readonly seed: number;   // fixed per fixture ⇒ regen determinism
  /** Drive from createInitialState(seed) to the waypoint via REAL
   *  intents (reduce) — focusedOptimalIntent by default, with a stop
   *  predicate; applyGrindFight allowed exactly where t0-arc uses it.
   *  NO state surgery: no field pokes, no forced flags. */
  readonly play: (s0: GameState) => GameState;
  /** Waypoint truths, asserted at GEN time and at TEST time. Written on
   *  STABLE invariants (rung id, location, flag booleans, band names,
   *  predicate results) — not exact numerics — so a routine balance
   *  tweak doesn't cry wolf. */
  readonly expect: (s: GameState) => void; // throws on miss
}
```

Shared drive helpers (extracted once, reused by every spec):
`runPolicy(s, stop)` — the t0-arc loop that PEEKS at the next
`focusedOptimalIntent` and stops BEFORE dispatching when
`stop(s, next)` fires (this is how "pre-wolf" stops one click short of
`face_wolf`); `grindUntil(s, mobId, stop)` — walk to the mob's node via
real `move_to` hops, then `fight` until the stop predicate holds.

**`src/scripts/gen-fixtures.ts`** (gen-docs idiom): for each spec —
`createInitialState(seed)` → `play` → `expect` (fail loudly if a spec
no longer reaches its own waypoint) → `makeEnvelope(state, 0, 0)` →
`JSON.stringify(env, null, 2) + '\n'` → write
`src/fixtures/saves/<name>.json`. Bare run writes; `--check`
regenerates into memory and diffs byte-for-byte. npm scripts:
`fixtures:regen` / `fixtures:check`.

**Is regen byte-stable? Yes — analyzed:** the core is pure (no
`Date.now`/`Math.random`; the RNG is seed + integer cursors inside
`GameState`); `focusedOptimalIntent` is a pure function of state; the
only nondeterminism in the save path is `SaveManager`'s injected clock
and counter, which the generator bypasses with `makeEnvelope(state, 0,
0)`; `JSON.stringify` key order is insertion order, fixed by the code
path; number formatting is spec-defined. So same code ⇒ same bytes, and
the `--check` drift gate is sound. (Ph1's DoD proves it empirically:
regen twice, `git diff --exit-code`.)

### 2.3 The scenario set v1 — six waypoints, each earning its place

| name | waypoint (stop condition) | why it earns a slot |
|---|---|---|
| `fresh-R3-pre-wolf` | R2, rung meter full, standing at the kura, `first-fight-survived` false — stopped the click before `face_wolf` | THE first-fight taste-check (the game's scripted combat debut, on the R3 doorstep): tuning wolf feel, the face-the-wolf CTA, and combat-reveal copy currently costs a full cold-open climb every time |
| `rung-beat-ready` | the first ready rung beat, stopped before `begin_rung_beat` | reviews the D-110 rung-up STORY-BEAT VN modal (trigger affordance + scene + choice) — a one-shot surface that is otherwise consumed the moment you pass it |
| `post-loss-broke` | drive to R3 with carried coin/rice, walk to the bandit, lose (the m2 seed-robust curve: low-level vs bandit ≈ 0.00), stop after the rout: `hp === SETBACK_HP`, coin/rice bled, auto stopped | the post-loss economy FEEL (D-113): does a setback read fair? Is the kura-sheltering legible? The rebuild loop's first minutes are the game's emotional low point and nobody replays into it voluntarily |
| `worn-weapon-no-wood` | fight the monkey until `durabilityBand` = Battered with `wood < REPAIR_WOOD_COST` (if wood ≥ 5 mid-loop, `repair_weapon` — a real intent — spends it and the loop continues) | the repair-loop bind: worn-weapon win-rate cliff, the repair CTA, the auto-combat weapon-broken stop — the exact state the auto-loop code (`main.ts:256-267`) worries about |
| `pre-ascension` | the full t0-arc drive, stopped when `ascensionAvailable(s)` is true (tier 0, Estate EXCELLENT, capstone flag set) | the T0→T1 ceremony review with a REAL earned history (real log, real resources) — unlike `jumpToAscension`, which forges flags onto whatever state you had; also the cockpit's "endgame economy" start-state |
| `wealthy-idler` | Phase 2, labour + `sell_rice` + `deposit` until banked coin ≥ a balance-derived threshold (e.g. 2× the dearest `ESTATE_STAGES` cost — no magic numbers), idle at the kura | rich-idle screenshots and "does wealth still have sinks?" tuning; loads genuinely idle for free — F32 resets in-flight automation on every load |

Names are the `__qa` keys, the filenames, and the panel labels — one
namespace. The set is deliberately small; growth comes from capture
graduation (§2.6) and the persona bots, not speculation.

### 2.4 Load UX — one click, and the human's run is never lost

- **`__qa.loadFixture(name)`** (DEV-only, next to `load`): backs up the
  CURRENT run to the F96 slot first (`save.backup(state)` — same as
  `newGame`), then `importState(toBase64(JSON.stringify(env)))`, then
  the standard post-import sequence `__qa.load` already does
  (`prev = null`, `revealPass`, render). Returns the `LoadResult` so
  scripts can assert. **`__qa.fixtures()`** lists `{name, blurb}`.
- **DEV panel: a third sub-tab "Scenarios"** beside Settings/Variants
  (`dev.ts` tab bar). One row per spec: name (bold mono), blurb
  (muted 11px), a `Load` button. Loading calls `qa.loadFixture(name)`
  and then `enableRestore()` — the existing "↩ last backup" button is
  the undo, so a playtest can never destroy the human's real save.
  `DevQa` grows `loadFixture` + a fixtures list (or the panel imports
  `FIXTURE_SPECS` directly — it is already a DEV-stripped module).
- **`?fixture=<name>` URL param** (DEV branch of `main.ts`, F18 idiom):
  boot → loadFixture. This is what makes `qa-shots.mjs`, the
  `capture-game-states` skill, and the cockpit one-liners:
  `page.goto('/?fixture=pre-ascension')`.
- Because `importState` ADOPTS the fixture as the newest save, a reload
  keeps you in the scenario — correct for a playtest session; the
  backup slot is the way back.

### 2.5 Teeth

**(a) The round-trip vitest test** (`src/fixtures/fixtures.test.ts`) —
IN the suite (it is milliseconds: parse + `validateEnvelope` + assert;
no simulation):

- every `saves/*.json` on disk has a spec and vice versa (registry ↔
  disk parity — an orphan or missing fixture is RED);
- each fixture loads through `validateEnvelope` (which runs the
  migration chain when `schemaVersion` lags) with `ok: true`;
- each loaded state passes its spec's `expect` — rung, location, hp,
  flags, band names.

**Soundness analysis:** `validate.ts` is deliberately additive-tolerant
(absent fields default silently), so "it validates" alone would be a
weak guard — a broken/missing migration step often still validates.
The `expect` assertions are what give this test teeth: they fail when a
defaulted field lands the state OFF its waypoint. Timing: at a schema
bump the committed fixtures are one version behind until regen, so this
test exercises the REAL chain on real old envelopes at exactly the
commit that bumps the schema — the loud failure you want. (The
synthetic per-step chain coverage stays `migrate.test.ts`'s job.)

**(b) The `fixtures:check` drift gate** — byte-stability (§2.2) makes
it possible; whether it joins the verify roster is a budget question.
Cost = regenerating all six, dominated by the two full-arc sims
(`pre-ascension`, `wealthy-idler`) — the same work `t0-arc.test.ts`
already does twice inside the vitest gate, so the estimate is ~1–2 s,
under the vitest critical path in the parallel roster. **Default:
add it to `verify-run.ts` GATES, then prove it with `verify:budget` in
Ph3's DoD; if the median breaches 5 s, demote it to the `gh-pages.sh`
deploy flow** (the established rung for slower checks — the capture
plan's §5.4 precedent). Note the contract it creates: any
engine/balance/copy change that moves the driven arc REDs the gate
until `fixtures:regen` — the same regen-on-change contract as
`gen-docs --check` / `pacing --check`, so it is house-normal, not
wolf-crying.

**(c) Strip gate — fixtures must not ship.** `src/fixtures/index.ts`
exports `FIXTURES_SENTINEL = '__KAMI_FIXTURES__'` and the Scenarios
pane stamps it on its DOM node (so it appears in the bundle iff the
module does). `gh-pages.sh` step 1b's marker loop gains
`"__KAMI_FIXTURES__"` and one fixture name (`fresh-R3-pre-wolf`) —
belt and braces: the second catches an emitted JSON chunk even if the
registry were somehow dropped while its imports survived. Mechanism of
absence: the registry is referenced only from `import.meta.env.DEV`
branches, the same fold-to-dead-code + tree-shake path that already
strips `ui/dev.ts` — and the grep PROVES it per deploy rather than
trusting it (R2).

### 2.6 Sibling-plan wiring (cross-reference, don't duplicate)

- **Balance cockpit (`fable-process-F7-balance-cockpit.md`):** fixtures
  ARE its known start-states — it should load them via
  `__qa.loadFixture` / `?fixture=`, never grow a second snapshot
  mechanism.
- **Balance-sim gates (`fable-process-F4-balance-sim-gates.md`):** when
  `src/sim/personas.ts` lands, a spec's `play` can be built from a
  persona policy instead of `focusedOptimalIntent` (e.g. an
  `idler-day-30` fixture). The spec model above needs no change —
  `play` is just a function.
- **Playtest capture inbox (`fable-process-F3-playtest-capture-inbox.md`)
  — the graduation flow:** a drained capture whose moment keeps being
  useful graduates to a fixture by writing a SPEC that reproduces the
  property (policy + stop condition + expects), then `fixtures:regen` —
  NEVER by committing the captured blob (a blob is hand-frozen data,
  exactly what the no-hand-maintained norm forbids; blobs stay one-off
  `__qa.load` payloads). Add this as an explicit triage outcome in
  `/drain-inbox` when that plan builds.

## 4 · Risks (each with a proposed default)

1. **Byte-stability breaks subtly** (a content registry reorders, a
   float formats differently). The `--check` gate catches it the same
   day; if a genuinely unstable source appears, fall back to
   semantic-equality check (parse + deep-equal) and note it — never
   hand-patch a fixture.
2. **Regen cost grows with the arc** (the intended ~85-min climb will
   raise intent counts as balance retunes). Guarded loops + the
   `verify:budget` measurement; if `fixtures:check` outgrows the
   budget, it demotes to the deploy flow — the round-trip test (cheap)
   stays in verify regardless.
3. **Regen churn / large diffs on every balance tweak.** Accepted —
   it is the same contract as every generated artifact here; diffs are
   reviewable JSON by design (§2.1), and the `expect` assertions are
   the actual review, not eyeballing 300 log lines.
4. **A panel misclick nukes the human's run.** Backup-first is
   structural (`loadFixture` cannot load without writing the F96 slot);
   the restore button lights up on load.
5. **Additive validation masks a missing migration** (§2.5a). The
   waypoint `expect`s are the mitigation; keep them on invariants that
   a defaulted field WOULD move (flags, rung, location, bands).
6. **Fixture-set rot** (specs nobody loads). The set is small and
   named; the qa-playtesting doc lists them; pruning a stale spec is a
   one-commit delete of spec + JSON (registry-parity test enforces the
   pair).
7. **`fresh-R3-pre-wolf` naming vs mechanics** (the wolf is faced AT
   R2). Keep the human-given name; the blurb states it plainly ("R2,
   wolf-ready — the fight that opens R3") so the label never misleads.

## 5 · Open questions (proposed defaults)

1. **Does `fixtures:check` sit in verify or the deploy flow?** Default:
   verify, demote on measured breach (§2.5b) — decided by Ph3's
   `verify:budget` number, not by taste.
2. **`?fixture=` in v1?** Default: yes, Ph2 — it is ~6 lines on the
   F18 idiom and it is what makes the cockpit/screenshot one-liners
   work.
3. **`wealthy-idler` threshold source?** Default: derive from
   `balance` constants (2× the dearest estate stage); revisit when the
   cockpit defines its own "rich" band.
4. **Should `loadFixture` exist in the prod console for player
   support?** Default: no — DEV-only, stripped; player-facing restore
   stays the Settings export/import save code.
